# Extend UAsset Parser

Preserve the parser as a generic saved-package reader while exposing small, versioned projections for
product domains. Treat Unreal as the behavioral oracle, not as the parser's internal architecture.

## Load project context

Before changing code:

1. Read `docs/vision-and-architecture.md`.
2. Read `docs/engineering/README.md` and its testing guide.
3. Read `docs/research/uasset-parser-roadmap.md`.
4. Inspect the existing parser model, `inspect` output, relevant projections, and fixtures.
5. Verify unfamiliar Unreal behavior against the configured engine source and a real fixture.

## Keep three layers distinct

Model new support as:

```text
package bytes
  -> rich internal decoded package model
  -> generic inspection evidence
  -> asset/domain projection
```

- Keep package structure, property tags, offsets, references, raw evidence, and decode failures in the
  internal model or generic inspection output.
- Do not reshape the core parser around one Unreal plugin response or product workflow.
- Project from the internal decoded model, not from another serialized output DTO. If existing code
  transforms `InspectOutput` into a domain output, avoid extending that coupling and improve the
  boundary when the task reasonably permits.
- Give each public process boundary a versioned, language-neutral schema with explicit limits and
  partial/unsupported results.
- Let domain consumers depend on normalized projections rather than parser implementation details.

## Decide whether to reuse a contract

Reuse one shared contract when saved-package and Unreal producers represent the same domain concept,
such as a DataTable snapshot. Include authority and provenance so saved disk state cannot be confused
with live editor memory.

Create a separate projection when the consumer needs different semantics. Do not force unrelated
assets into the authoring snapshot merely because comparison becomes syntactically convenient.

Keep generic inspection separate when evidence has no truthful Unreal or domain equivalent, including
package versions, offsets, imports/exports, tail bytes, and parser diagnostics.

## Add an asset type

1. State the product question the asset support answers.
2. Characterize representative assets generated by the generic Unreal fixture.
3. Identify the cheapest truthful layer for every required fact: package bytes, Unreal reflection,
   editor-only API, or domain derivation.
4. Extend the internal decoded model and codecs without losing unknown or unsupported evidence.
5. Add or extend generic inspection only when the evidence is broadly useful.
6. Add a narrow domain projection with an authoritative schema when a product consumes it.
7. Normalize producer-specific differences before comparison.
8. Test parsing, projection, protocol conformance, malformed input, limits, and Unreal parity at their
   cheapest truthful layers.

Do not introduce project-specific paths, schemas, assets, or assumptions.

## Compare with Unreal

Prefer two independent producers that converge on canonical semantic evidence:

```text
Rust saved-package decode ----> domain projection --+
                                                    +--> canonicalize --> structural diff
Unreal load/reflection --------> domain projection --+
```

Use a headless commandlet for deterministic package-load and reflection conformance. Use Remote
Control or the companion plugin for live memory, editor-only behavior, mutation, save, and reload.

Exclude provenance-only differences such as authority, producer identity, session identity, and
diagnostic envelopes from semantic equality. Canonicalize unordered sets and maps. Compare ordered
rows, arrays, and parent precedence as ordered data. Prefer a structural diff during development;
fingerprints are a compact assertion, not sufficient mismatch diagnostics.

Classify differences explicitly:

- semantic mismatch;
- parser unsupported or partial;
- unavailable through Unreal reflection;
- expected authority/provenance difference;
- intentional normalization.

Do not make Unreal emit the parser's internal representation. That would couple the oracle to the
implementation and can allow matching mistakes to hide each other.

## Fixture and test policy

- Generate fixtures through Unreal and commit small `.uasset` files with reviewable source inputs.
- Cover one minimal asset plus boundary values, nested/container values, references, and a deliberately
  unsupported value where applicable.
- Run ordinary parser tests without requiring Unreal.
- Gate regeneration and real-Unreal conformance separately so CI can run them where UE is installed.
- When Unreal regenerates evidence, reload assets in a fresh process before accepting results.
- Add malformed-package regression tests separately; valid Unreal fixtures do not exercise hostile
  input handling.

After edits, run the focused tests and then `pnpm check`.
